## Codebase Patterns
- `brand-lint` is an inline bash one-liner in `unit/package.json` `scripts.brand-lint` ŌĆö a DENY-list (greps for banned `bg-white|bg-gray-|text-gray-|border-gray-|text-black|bg-black|font-bold|font-medium|font-nunito-bold|font-lora-bold|text-xs|text-lg|text-xl|text-4xl`). Adding new positively-named brand tokens to `tailwind.config.js` does NOT trip it.
- There is NO `npm run typecheck`. Use `npx tsc --noEmit` from `unit/`.
- This project root is `UNIT-PRoject/` (no trailing space). The git repo lives at `unit/.git`, NOT at the parent ŌĆö run git commands from `unit/` or use `git -C unit`.
- Tailwind brand color tokens live in `unit/tailwind.config.js` under `theme.extend.colors.brand`. Class names are auto-derived (e.g. `cloud` ŌåÆ `bg-brand-cloud`, `'ink-muted'` ŌåÆ `text-brand-ink-muted`).
- Brand contract is documented in `unit/CLAUDE.md` "Brand Theming" section ŌĆö keep it in sync with `tailwind.config.js` whenever palette/typography changes.
- When extending DB-row types in `unit/services/*.ts`, also update the local `*_COLUMNS` constant in the same file (e.g. `PROPERTY_COLUMNS` in `properties.ts`). Forgetting this means new fields silently come back undefined at runtime even though TS thinks they exist.
- When adding required fields to a row type used by a service `create()`/`update()` Omit signature, audit every caller ŌĆö TS will catch them but the fix is to pass an explicit `null` for nullable DB columns rather than loosen the Omit (keeps the schema honest at the boundary).
- Discriminator helpers for sum-typed rows (e.g. `PromotionKind`, `getPromotionKind(p)`) belong as exported functions next to the row type, NOT as an extra field on the row type ŌĆö Supabase `select('*')` won't populate computed fields and adding them to the type creates lying shapes.
- External-API helpers that are called from both UI hot paths and batch scripts (e.g. `services/geocoding.ts`) should serialize through a process-local throttle (single `lastRequestAt` timestamp + setTimeout wait) and NEVER throw ŌĆö return `null` and `console.warn` so callers can store NULL and retry later.
- Standalone Node scripts under `unit/scripts/` cannot import `services/supabase.ts` because that pulls in `@react-native-async-storage/async-storage`. Construct a fresh `createClient(url, SUPABASE_SERVICE_ROLE_KEY, { auth: { persistSession: false } })` inside the script. Service-role key is required for any RLS-bypassing batch update.
- Jest tests for services that issue both a `.eq().single()` query AND a list `.select()` on the same table need to mock `select()` per-call: `mockImplementationOnce(() => mockChain)` for the chained call, then `mockImplementationOnce(() => Promise.resolve({data, error}))` for the awaited list call. The list path resolves directly off `select()` because there is no `.single()`.
- For pure-math helpers (haversine, geometry, etc.) export the function alongside the service that uses it so tests can exercise the math directly without hitting the supabase mock at all. Kept `haversineMiles` exported even though only `getNearbyPropertyIds` consumes it internally ŌĆö makes Daytona-cluster math testable as 4 trivial assertions.
- For aggregator services that issue multiple parallel queries on different tables (e.g. activity feed across promotions/posts/businesses/notifications), prefer a per-table thenable mock keyed off the `from(table)` argument: a single `tableData: Record<string, unknown[]>` map plus a `from = jest.fn((t) => ({...chain, then: r => r({data: tableData[t] ?? [], error: null})}))` lets each test set canned data per table and lets `expect(from.mock.calls.map(c => c[0]))` directly verify the no-N+1 invariant. Much cleaner than chaining `mockImplementationOnce` per call.
- React Query keys for hooks that take an array argument should serialize the array into the key (`propertyIds.join(',')`) ŌĆö TanStack uses structural equality on the key, but a stable string segment is more predictable across re-renders and cheaper to hash. Pattern: `['activityFeed', propertyIds.join(','), limit]`.

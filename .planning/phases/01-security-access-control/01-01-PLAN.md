    **Section 1 — profiles table (per D-03):**
    ```sql
    create table profiles (
      id uuid primary key references auth.users(id) on delete cascade,
      role text not null default 'tenant' check (role in ('tenant', 'landlord')),
      property_ids uuid[] default '{}',
      email text,
      created_at timestamptz default now()
    );
    alter table profiles enable row level security;
    create policy "Users can read own profile"
      on profiles for select to authenticated
      using (id = auth.uid());
    create policy "Service role can manage profiles"
      on profiles for all to service_role
      using (true) with check (true);
    ```

    **Section 2 — is_landlord() helper function:**
    ```sql
    create or replace function is_landlord()
    returns boolean
    language sql security definer stable
    as $$
      select exists (
        select 1 from profiles
        where id = auth.uid()
        and role = 'landlord'
      );
    $$;
    ```

    **Section 3 — landlord_property_ids() helper function (AUTH-03 property isolation):**

    CRITICAL for AUTH-03: Create a second helper function that returns the current landlord's property_ids array. This is used in all financial table RLS policies to enforce per-property data isolation — not just role checking.

    ```sql
    create or replace function landlord_property_ids()
    returns uuid[]
    language sql security definer stable
    as $$
      select coalesce(
        (select property_ids from profiles where id = auth.uid() and role = 'landlord'),
        '{}'::uuid[]
      );
    $$;
    ```

    Why a separate function: Inlining the subquery in every policy would work, but a `security definer` function avoids repeating the subquery 20+ times and ensures consistent behavior. The function returns an empty array for non-landlords (via coalesce), which means the `= ANY(...)` check naturally returns false.

    **Section 4 — Drop ALL existing placeholder financial policies and replace with property-scoped policies:**
    Drop these exact policy names (from 001_initial_schema.sql lines 296-338):
    - "Leases viewable by authenticated users", "Leases writable by authenticated users", "Leases updatable by authenticated users", "Leases deletable by authenticated users"
    - "Recurring payments viewable", "Recurring payments writable", "Recurring payments updatable", "Recurring payments deletable"
    - "Invoices viewable", "Invoices writable", "Invoices updatable", "Invoices deletable"
    - "Expenses viewable", "Expenses writable", "Expenses updatable", "Expenses deletable"
    - "Payments viewable", "Payments writable"

    Replace with **property-scoped** landlord policies for ALL 5 financial tables. Each policy checks BOTH `is_landlord()` AND that the row's `property_id` is in the landlord's `property_ids` array. This is the AUTH-03 fix — a landlord with property_ids=[A] CANNOT read/write financial data for property B.

    Each table gets 4 policies (select, insert, update, delete — except payments which needs select, insert, update):
    ```sql
    -- Pattern per table (repeat for leases, recurring_payments, invoices, expenses, payments):
    create policy "{Table} viewable by landlords"
      on {table} for select to authenticated
      using (is_landlord() and property_id = any(landlord_property_ids()));
    create policy "{Table} writable by landlords"
      on {table} for insert to authenticated
      with check (is_landlord() and property_id = any(landlord_property_ids()));
    create policy "{Table} updatable by landlords"
      on {table} for update to authenticated
      using (is_landlord() and property_id = any(landlord_property_ids()));
    create policy "{Table} deletable by landlords"
      on {table} for delete to authenticated
      using (is_landlord() and property_id = any(landlord_property_ids()));
    ```

    For payments table: only create select, insert, update policies (no delete — consistent with original schema which only had "Payments viewable" and "Payments writable"). Actually, add all 4 for consistency with the other financial tables — delete is also landlord-scoped.

    **Section 5 — landlord_code field neutralization (per D-06, AUTH-06):**

    ```sql
    update properties set landlord_code = null;

    create or replace function prevent_landlord_code_write()
    returns trigger language plpgsql as $$
    begin
      new.landlord_code = null;
      return new;
    end;
    $$;

    create trigger no_landlord_code
      before insert or update on properties
      for each row execute function prevent_landlord_code_write();
    ```

    **Section 6 — audit_log table (per D-15):**

    ```sql
    create table audit_log (
      id uuid primary key default uuid_generate_v4(),
      entity_type text not null,
      entity_id uuid not null,
      action text not null,
      old_value jsonb,
      new_value jsonb,

      performed_at timestamptz default now()

    create policy "Landlords can insert audit entries"
      on audit_log for insert to authenticated
      with check (is_landlord());
    create policy "Landlords can read audit entries"
    ```
